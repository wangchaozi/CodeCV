# RAG（检索增强生成）原理详解与面试高频考点

## 一、什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将**信息检索**与**大语言模型生成**结合的 AI 架构。

> 简单一句话：**先检索，再生成**。让 LLM 在回答问题前，先从知识库里找到相关资料，再基于资料作答。

### 为什么需要 RAG？

LLM（如 GPT、Gemini）本身存在三大核心缺陷：

| 缺陷 | 问题描述 | RAG 如何解决 |
|------|---------|------------|
| **知识截止** | 训练数据有截止日期，不知道最新信息 | 从实时更新的知识库检索 |
| **幻觉（Hallucination）** | 模型会自信地编造不存在的事实 | 给定参考文档，减少凭空生成 |
| **上下文窗口有限** | 无法把整个知识库塞进 prompt | 只检索最相关的片段注入 |

## 二、RAG 的完整工作流程

```
用户提问
   │
   ▼
┌─────────────────────────────────────────────────────────────────┐
│                        离线阶段（建索引）                          │
│  原始文档 → 文本分块（Chunking）→ 向量化（Embedding）→ 向量数据库   │
└─────────────────────────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────────────────────────┐
│                        在线阶段（检索 + 生成）                      │
│  问题向量化 → 相似度检索 Top-K → 拼接上下文 → LLM 生成答案          │
└─────────────────────────────────────────────────────────────────┘
   │
   ▼
最终答案（带来源引用）
```

### 详细步骤拆解

#### 阶段一：文档入库（离线）

```
① 文档加载（PDF / TXT / MD / DOCX）
② 文本清洗（去除无效字符、表格处理）
③ 文本分块（Chunking）
   - 固定长度分块：每 512 Token 切一段
   - 滑动窗口：块间有重叠（如 100 Token 重叠），防止关键信息被截断
   - 语义分块：按段落/章节切分（更智能）
④ 向量化（Embedding）
   - 每个文本块通过 Embedding 模型转为高维向量
   - 如 text-embedding-3-small 输出 1536 维向量
⑤ 存入向量数据库（如 Milvus / Pinecone / pgvector）
```

#### 阶段二：检索（在线）

```
① 用户问题同样经过 Embedding 模型转为向量
② 在向量数据库中计算相似度（余弦相似度 / 点积）
③ 返回 Top-K 个最相关的文本块
④ 可选：Reranking（重排序，用更强模型精排结果）
```

#### 阶段三：生成（在线）

```
① 将检索到的 Top-K 文本块拼接为 Context
② 构造 Prompt：
   System: 你是XX助手，根据以下资料回答问题
   Context: [检索到的相关文档片段]
   User: [用户原始问题]
③ 送入 LLM 生成最终回答
④ 可附带来源引用（溯源）
```

## 三、核心技术组件

### 1. Embedding 模型

将文本转为稠密向量，语义相近的文本向量距离近。

```
"JavaScript 是什么" → [0.12, -0.34, 0.89, ...]  (768/1536 维)
"JS 入门介绍"       → [0.11, -0.35, 0.87, ...]  (距离很近 ✓)
"Python 教程"       → [0.56,  0.12, -0.23, ...]  (距离较远 ✗)
```

常见 Embedding 模型：
- **OpenAI** text-embedding-3-small / large
- **Google** text-embedding-004（768 维）
- **开源** BGE-M3、E5-large、nomic-embed

### 2. 向量相似度算法

```python
# 余弦相似度（最常用）
cos_sim = dot(A, B) / (|A| * |B|)  # 范围 [-1, 1]，越大越相似

# 欧氏距离（L2）
l2_dist = sqrt(sum((A - B)²))  # 越小越相似

# 点积（内积）
dot_product = sum(A * B)  # 向量已归一化时等同于余弦相似度
```

### 3. 向量数据库对比

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| **Milvus** | 开源、高性能、分布式 | 大规模生产 |
| **Pinecone** | 托管服务、开箱即用 | 快速原型 |
| **Weaviate** | 内置混合检索 | 语义+关键词结合 |
| **pgvector** | PostgreSQL 插件 | 已有 PG 的项目 |
| **FAISS** | Meta 开源、纯内存 | 研究/离线批处理 |
| **Chroma** | 轻量、本地开发友好 | 小项目/开发调试 |

### 4. 分块策略（Chunking）

```
文档 ──────────────────────────────────────────────────────
│  第一段：RAG 简介（300字）  │  第二段：工作原理（400字）  │
└───────────────────────────────────────────────────────────

固定分块（stride=200, overlap=50）：
  块1: [0-200]
  块2: [150-350]   ← 50字重叠，避免关键信息被截断
  块3: [300-500]
```

## 四、RAG 的进阶优化技术

### 1. HyDE（假设文档嵌入）
先让 LLM 生成一个假设性答案，再用这个答案去检索（比原始问题检索效果更好）。

### 2. Multi-Query
将用户问题改写为多个不同角度的子问题，分别检索，合并结果。

### 3. Reranking（重排序）
用 Cross-Encoder 对检索到的候选块精排，提升准确率（代价是速度变慢）。

### 4. Self-RAG
让模型自主决定：是否需要检索？检索到的内容是否相关？答案是否需要修正？

### 5. Hybrid Search（混合检索）
向量检索 + 关键词检索（BM25）结合，用 RRF（倒数排名融合）合并结果。

## 五、本项目 RAG 实现

```
用户在问答助手中提问
        │
        ▼
  问题向量化（text-embedding-004）
        │ 失败时降级
        ▼
  相似度检索 or 全文关键词检索（ILIKE）
        │
        ▼
  取 Top-5 文本块作为 Context
        │
        ▼
  Gemini streamGenerateContent（SSE 流式）
  systemInstruction = RAG系统提示 + Context
        │
        ▼
  逐 Token 推送给前端（Server-Sent Events）
        │
        ▼
  前端实时渲染 + 显示来源引用
```

技术栈：
- **后端**：NestJS + TypeORM + PostgreSQL
- **向量存储**：PostgreSQL JSONB（embedding 列）
- **相似度计算**：Node.js 内存余弦相似度
- **降级方案**：PostgreSQL ILIKE 全文搜索
- **LLM**：Gemini via SSE 流式接口

## 六、面试高频考点

### Q1：RAG 和 Fine-tuning（微调）有什么区别？选哪个？

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 实时，改数据库即可 | 需重新训练，成本高 |
| 数据量要求 | 无需标注数据 | 需要大量高质量标注数据 |
| 可解释性 | 高（可溯源） | 低（黑盒） |
| 适合场景 | 知识密集型问答、企业私有数据 | 特定风格/任务能力增强 |
| 成本 | 低（推理时多一次检索） | 高（需训练 GPU）|

**结论**：知识型任务选 RAG，行为/风格调整选 Fine-tuning，两者可结合。

---

### Q2：如何评估 RAG 系统的质量？

**检索质量指标：**
- **Recall@K**：Top-K 结果中包含正确答案的比例
- **MRR（Mean Reciprocal Rank）**：正确答案排在第几位的倒数均值
- **NDCG**：归一化折损累积增益

**生成质量指标：**
- **Faithfulness（忠实度）**：答案是否基于检索内容，不编造
- **Answer Relevancy（相关性）**：答案是否回应了用户问题
- **Context Precision/Recall**：检索上下文的精确率/召回率

工具：**RAGAS**（专门评估 RAG 的框架）

---

### Q3：RAG 中的 Chunk Size 如何选择？

| Chunk 大小 | 优点 | 缺点 |
|-----------|------|------|
| 小（128-256 Token）| 精准，噪声少 | 可能缺失完整上下文 |
| 大（512-1024 Token）| 上下文完整 | 噪声多，相似度被稀释 |
| **推荐：256-512 Token + 50-100 Token 重叠** | | |

实践技巧：父子分块（Parent-Child Chunking）—— 用小块检索、大块送入 LLM。

---

### Q4：如何解决 RAG 的幻觉问题？

1. **来源引用**：强制要求模型注明信息来源
2. **忠实度检查**：用 NLI 模型验证答案是否能从 Context 推导
3. **Grounding 提示**：Prompt 中明确要求"仅基于提供的资料回答"
4. **置信度阈值**：检索相似度低于阈值时，回答"知识库中无相关内容"
5. **Self-RAG**：让模型评估自己的答案是否有依据

---

### Q5：余弦相似度 vs 欧氏距离，RAG 中用哪个？

**通常用余弦相似度**，原因：
- 对向量长度不敏感，只关注方向（语义）
- Embedding 模型输出通常已归一化，余弦 = 点积，计算更快
- 文档长短不同时，余弦比 L2 更公平

---

### Q6：RAG 系统如何处理多跳问题（Multi-hop）？

多跳问题需要推理多个文档才能得出答案，如："A 公司的 CEO 毕业于哪所大学？"

解决方案：
1. **迭代检索**：第一跳找到 CEO 是谁，第二跳再找该人学历
2. **Multi-Query**：将问题分解为子问题分别检索
3. **Graph RAG**：构建知识图谱，沿关系路径检索
4. **ReAct 模式**：让 LLM 在推理过程中决定何时检索、检索什么

---

### Q7：向量数据库的 ANN（近似最近邻）算法

精确 KNN 复杂度 O(n)，向量库用 ANN 加速：

| 算法 | 原理 | 代表实现 |
|------|------|---------|
| **HNSW** | 分层图结构，导航到最近邻 | Milvus、Weaviate |
| **IVF** | 倒排文件索引，先聚类再搜索 | FAISS |
| **PQ（乘积量化）** | 压缩向量降低内存和计算 | FAISS |
| **ScaNN** | Google 提出的各向异性量化 | Google ScaNN |

---

### Q8：RAG 的上下文窗口用完了怎么办？

当检索到的 chunks 超过模型上下文限制时：
1. **减少 topK**：只取相关性最高的 3-5 个块
2. **Map-Reduce**：分批让模型处理每个块，再汇总
3. **Refine 策略**：逐块迭代，将前一块的答案作为下一块的基础
4. **长上下文模型**：如 Gemini 1.5 Pro（100万 Token 窗口）直接塞入
